# AshTypescript: стоит ли использовать и как это сравнивается с альтернативами

## Что такое AshTypescript

AshTypescript — это расширение для Ash Framework (Elixir), которое автоматически генерирует TypeScript-типы и RPC-функции из ваших Ash ресурсов. Вы описываете модель на Elixir один раз, запускаете `mix ash_typescript.codegen`, и получаете полностью типизированный клиент для фронтенда.

Мы прошли полный цикл: создали ресурс `Book`, настроили домен с `AshTypescript.Rpc`, подключили Next.js frontend, добавили Phoenix Channels с Presence для real-time. Вот что из этого вышло.

---

## Плюсы

### 1. Единый источник правды

Вся схема данных живёт в Elixir. Добавил поле `rating` к книге — после `mix ash_typescript.codegen` TypeScript сразу знает о нём и подскажет, где его не хватает. Это принципиально отличается от REST, где бэк и фронт часто расходятся молча.

```elixir
# Добавил в Elixir
attribute :rating, :integer do
  public? true
end
```

```typescript
// Фронт сразу видит
type BookResourceSchema = {
  // ...
  rating: number | null;
}
```

### 2. Нет ручного написания API

Нет контроллеров, нет сериализаторов, нет документации на эндпоинты. Вы описываете действия (`create`, `update`, `read`) один раз в ресурсе, прописываете их в `typescript_rpc` блоке — и они появляются как функции на фронте.

### 3. Типизированные возвращаемые значения зависят от запрошенных полей

Это реально впечатляет. Если вы запрашиваете только `["id", "title"]`, TypeScript знает, что в ответе будет `{ id: string } & { title: string }` — без лишних полей. Это ближе к GraphQL, чем к обычному REST.

```typescript
const result = await listBooks({ fields: ["id", "title"] as const });
// result.data — Array<{ id: string } & { title: string }>
// result.data[0].author — ошибка компилятора!
```

### 4. Real-time из коробки

`generate_phx_channel_rpc_actions: true` генерирует WebSocket-версии тех же функций. Phoenix Channels + `Ash.Notifier.PubSub` + Phoenix Presence — и у вас живой список с счётчиком онлайн-пользователей за пару часов работы.

### 5. Меньше бойлерплейта, чем в GraphQL

В GraphQL нужно: схема + резолверы + N+1 решения (DataLoader) + Apollo/urql на фронте. В AshTypescript: описал ресурс, добавил `rpc_action` — готово.

## Минусы

### 1. Жёсткая привязка к Ash и Elixir

Это не универсальное решение. Нет Ash — нет AshTypescript. Если вы хотите разделить бэкенд и фронтенд на разные команды или сменить бэкенд-стек, придётся переписывать.

### 2. Ручная синхронизация сгенерированного файла

`ash_generated.ts` нужно копировать в проект фронтенда руками (или настраивать CI). В монорепозитории это несущественно, но если у вас отдельный репозиторий для фронта — нужен скрипт или публикация в npm.

### 3. Сложные типы — источник боли

Мы столкнулись с `readonly tuple` проблемой, с тем, что TypeScript не может сузить generic discriminated union. Генерированный файл весит ~950 строк условных типов. Для опытного TypeScript-разработчика это нормально, но джуниора может напугать.

### 4. Молодая экосистема

Мало примеров, документация есть но неполная, некоторые паттерны нужно нащупывать самостоятельно (как мы и делали). Вопросы на StackOverflow почти не найдёшь.

### 5. Нет GraphQL-гибкости

В GraphQL фронтенд сам решает что запросить, в том числе по вложенным связям произвольной глубины. В AshTypescript вы явно декларируете RPC-экшены — нельзя произвольно попросить `book.author.books.reviews`.

## Сравнение с альтернативами

| Подход | Типизация | Бойлерплейт | Гибкость | Real-time | Привязка к стеку |
|---|---|---|---|---|---|
| **AshTypescript** | ⭐⭐⭐⭐⭐ | Низкий | Средняя | Из коробки | Только Ash |
| **REST + OpenAPI** | ⭐⭐⭐ | Средний | Высокая | Отдельно | Нет |
| **GraphQL** | ⭐⭐⭐⭐ | Высокий | Очень высокая | Отдельно | Нет |
| **tRPC** | ⭐⭐⭐⭐⭐ | Низкий | Средняя | Отдельно | Только Node.js |
| **Phoenix LiveView** | ⭐⭐ | Очень низкий | Низкая | Родная | Только Phoenix |

### REST + OpenAPI

Самый распространённый вариант. Плюс — стандарт индустрии, работает везде. Минус — типы часто устаревают, генерация из схемы часто даёт `any` там где не ожидаешь, и кто-то должен поддерживать документацию.

### GraphQL

Идеален когда фронтенд сложный, запросы нетривиальные, или несколько клиентов (web, mobile, партнёры). Дорого в поддержке: resolver hell, N+1, кеширование. Если у вас небольшое SPA с понятной моделью — это оверинжиниринг.

### tRPC

Ближайший аналог AshTypescript, но для Node.js. Те же идеи: типобезопасный RPC, нет ручного API. Разница: tRPC живёт внутри TypeScript монорепо и это его сила — никакого codegen, просто импортируешь типы напрямую. AshTypescript с Elixir на это не способен по природе — нужен codegen шаг. Если вы на Node.js — tRPC лучше. Если на Elixir/Ash — AshTypescript.

### Phoenix LiveView

А вот это отдельная история, заслуживающая отдельного разбора.

## LiveView: рендерим TSX компоненты на сервере?

Вы спросили об интересном сценарии: а что если фронтенд делает `.tsx` компоненты, а бэкенд их рендерит и отдаёт через сокеты? Это в точности то, что делает **Phoenix LiveView** — только с HEEx-шаблонами вместо TSX.

### Как работает LiveView

Компонент живёт на сервере (Elixir). Клиент получает первичный HTML, подключается по WebSocket. При любом событии (клик, ввод) сообщение летит на сервер, сервер пересчитывает состояние, вычисляет diff и отправляет только изменения обратно. Клиент применяет diff к DOM.

```
Клик в браузере → WebSocket → handle_event на Elixir → diff → WebSocket → патч DOM
```

### Когда LiveView лучше React + AshTypescript

**CRUD-приложения, дашборды, формы** — LiveView здесь выигрывает. Меньше кода, нет API-слоя, состояние в одном месте (сервер), real-time бесплатно. Вы написали бы нашу книжную страницу за 30 минут против нескольких часов.

**Маленькие команды** — один разработчик ведёт всё на Elixir. Не нужно переключаться между языками и парадигмами.

**Контент с частыми обновлениями** — чаты, ленты, мониторинг — LiveView дешевле чем поднимать отдельный WebSocket-слой руками.

### Когда React + AshTypescript лучше

**Богатый клиентский UI** — анимации, drag-and-drop, оффлайн-режим, сложная форма с instant-feedback. LiveView ограничен задержкой сети, React работает мгновенно.

**Мобильное приложение** — API нужен в любом случае. AshTypescript даёт те же функции для React Native.

**Большая фронтенд-команда** — React-разработчики не хотят учить HEEx и Elixir. Отдельный Next.js — их среда.

**Экосистема React** — TanStack Query, Radix UI, Framer Motion, тысячи компонентов. В LiveView этого нет.

### Про "TSX через сокеты"

Строго говоря, это не про TSX. Идея рендеринга компонентов на сервере и отправки их клиенту существует в нескольких вариантах:

- **React Server Components (RSC)** — именно то, что делает Next.js App Router. Компонент рендерится на сервере, клиент получает RSC payload, не HTML. Это не LiveView, потому что нет двусторонней связи — это просто SSR с сохранением клиентского состояния.

- **LiveView + React** — можно запустить React компоненты внутри LiveView через `phx-hook`. Бэкенд управляет данными через LiveView, фронтенд рисует сложные вещи через React. Гибридный подход, но сложный в поддержке.

- **Чистый LiveView** — HEEx компоненты, никакого TSX. Проще всего, меньше JS, максимальный контроль на сервере.

## Итог: стоит ли использовать AshTypescript?

**Да, если:**
- Вы уже используете Ash Framework
- Строите SPA или мобильное приложение с React/React Native
- Хотите типобезопасность от базы до интерфейса без GraphQL сложности
- Нужен real-time и хочется не писать его руками

**Нет, если:**
- Ваш бэкенд не на Ash/Elixir
- Строите простое CRUD-приложение — LiveView справится быстрее
- Команда фронтенда большая и независимая — GraphQL даст им больше свободы
- Нужна схема API для партнёров/клиентов — OpenAPI понятнее

**Нейтрально:**
- Если вы на Node.js — смотрите на tRPC, идея та же, интеграция лучше
- Если проект растёт — AshTypescript и GraphQL не противоречат друг другу, Ash умеет в `AshGraphql`

AshTypescript — это правильная идея в правильной экосистеме. Он делает именно то, что обещает: убирает ручной труд по поддержке API-контракта между Elixir и TypeScript. Если вы в этом стеке — используйте.
